GSA EA Analytics and Reporting

GSA EA Analytics and Reporting (GEAR) is a tool to provide commonly used strategic, business and technical information to users throughout GSA in order to provide insight into the common direction for GSAIT and to provide working groups with data that they need to do their jobs.


Sources of Data

The source of the data is the EA tool, System Architect. The EA team built a set of openly available web services to provide the data to this portal and to other users who might need direct access to the EA data. Currently, the tool is available to all users within the GSA firewall.

Business Application Rationalization

The General Services Administration’s (GSA’s) Enterprise Architecture (EA) program has developed an innovative approach to Business Application Rationalization, a business-focused, data-driven methodology to analyze the IT application portfolio and help guide IT investment decisions. This methodology is used to assess whether a business application should be tolerated, invested, migrated, or eliminated and provides a consistent approach to coordinate discussions between business and IT stakeholders to discuss the strategic direction for GSA’s IT investments. In concert with GSA’s enterprise governance boards, the EA team offers recommendations, which inform the determination to retire, consolidate, replace or modernize GSA’s business applications. You can view the current TIME report generated by the Business Application Rationalization process here.

TIME

Tolerate: There are three levels of tolerate.

  • T1 are considered in "keep the lights on" mode and only critical fixes are completed (i.e., resolve issues only if the business application cannot be used at all or without major workarounds or for significant security vulnerabilities). No DM&E. Only O&M
  • T2 are tolerated with enhancements. Mostly O&M with some DM&E
  • T3 are tolerated, are likely to be considered for migration in the next 1 to 3 years and a migration plan is needed. Mostly O&M with DM&E for migration planning

Invest: Business Application is being developed/upgraded to provide functionality due to new business requirement or elimination of current business applications. Must have an approved EBC or 50% or more of budget is DM&E. Business Applications will have implementation plans detailing the planned enhancements.

Migrate: There are two levels of migrate. Migration planning and funding to complete migration should be approved so the migration can begin in the current fiscal year

  • M1 are considered business applications whose technology/platform/hardware are being modernized; business applications satisfy drivers to move to a new platform or technology. Technology focused.
  • M2 are considered business applications whose capabilities are migrating to new solution; business application satisfy drivers including improved integration with other business applications. Technology/platform/hardware should be modernized as part of this migration. Business focused (could include technology changes).

Eliminate: Business Application is no longer needed. Elimination date is known. Data migration plan is known

GEAR Web Service Description

The GEAR team has built RESTful API services, accessible from within the GSA firewall, that provides authoritative data from System Architect. The service powers this web application, but can also be used by other processes or applications. All service responses are in JSON format. The reusable services are listed below, and can be examined in more detail by clicking on the links.


Business Applications

For the purposes of GEAR and the Enterprise Architecture, Business Applications are those larger business-supporting IT efforts. This list is NOT intended to represent every software application, rather the funded, business-facing projects. There are two built-in switches on the service that will return fewer Business Applications, but more details about the items returned. The default service call will return the entire list of applications, but not all of the fields will be populated. The Parameters link below will provide details on the different switches that are available. The "JSON Definitions" link will define what is meant by each field, as well as indicate which fields will be populated when a particular switch is used.


Base URL: https://ea.gsa.gov/eaweb/eaopen/eaopen.svc/applications/
Parameters
Parameter Description
.../applications/ No parameters on this API will return the complete list of applications, including websites and retired applications. The fields with values are marked in the table below.
.../applications/?ID=<integer> Using this switch will only return the single application, but will include more detail. Selecting an invalid ID will return a null response.
.../applications/?SSO=<string> Using this switch will return all the applications that match the provided SSO. Selecting an invalid SSO will return a null response. Two-letter offices are not considered valid entries for an SSO.
JSON Definition

In the "Description" column, (1) indicates that a single value will be returned, while a (m) indicates that multiple entries may be returned. Those multiple entries will be separated by commas, unless the field is an array.

IT Standards

The "IT Standards" API exports GSA's IT Standards Profile. For this purpose, IT Standards fit into two categories. This list includes the component parts for GSA's Business Applications. This includes such things as databases, web servers, application servers, etc. The IT Standards Profile also includes technologies that have been approved for installation on GSA provided laptop and desktop machines. This includes items like Adobe Acrobat and Microsoft Office products.

Base URL: https://ea.gsa.gov/eaweb/eaopen/eaopen.svc/it_standards/All
